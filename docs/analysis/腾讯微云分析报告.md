## 三、JS 反编译分析

### 3.1 核心技术栈

### 3.2 关键发现 1 — `fingerprint` Cookie 分析

**已确认**：`fingerprint` Cookie 在运行时存在（`document.cookie` 可见）。该 Cookie 由后端（TAPISIX / stgw）在下发页面时直接设置，**不是**前端 JS 生成的。

**推断来源**：
- stgw（`tgw_l7_route` 同源）在首次访问时通过 `Set-Cookie` 下发
- TAPISIX 网关层可能读取此 Cookie 用于会话绑定
- 值为浏览器/会话唯一标识（格式同设备 ID 风格）

**与腾讯视频指纹的异同**：

| 对比项 | 腾讯微云 `fingerprint` | 腾讯视频 `vendor-common.js` 指纹 |
|:-------|:----------------------|:---------------------------------|
| 来源 | **服务端 Set-Cookie** | 前端 JS 采集 Canvas/WebGL/Audio |
| 前端采集 | 无 | Canvas 32位哈希 + WebGL + Audio |
| 采集时机 | 首次 HTTP 响应 | SPA 加载后执行 |
| 隐身模式 | 不重置（Cookie 持久化） | ⚠️ 采集值可能变化 |
| 参与会话绑定 | 🟡 可能 | ✅ 确认绑定 |

> `fingerprint` Cookie 对隐身/普通模式迁移的影响**较小**（它是持久化 Cookie，导出后直接恢复），真正风险在于腾讯视频那种**前端采集的 Canvas 哈希**——微云没有这种机制。

### 3.3 关键发现 2 — Aegis 设备标识（`AEGIS_ID`）

腾讯微云使用 **Tencent Aegis** 监控 SDK 生成本地设备标识：

```javascript
// Aegis SDK 中 AEGIS_ID 生成逻辑
init: function(e) {
    try {
        var t = !0 !== e && e || window.localStorage.getItem("AEGIS_ID");
        t || (t = Me(), window.localStorage.setItem("AEGIS_ID", t));
        this.aid = t;
    } catch(e) {}
}
```

| 属性 | 说明 |
|:-----|:------|
| **Key** | `AEGIS_ID` |
| **存储位置** | localStorage |
| **生成方式** | UUID v4（`Me()` 函数） |
| **刷新时机** | 仅 localStorage 为空时生成 |
| **用途** | 监控/分析设备标识，非认证绑定点 |
| **隐身模式** | ❌ 每次刷新生成新 ID（localStorage 在隐身模式下隔离） |
| **对插件影响** | 🟡 低（不参与认证，仅影响监控） |

Aegis SDK 还尝试从 Cookie 中提取 QQ `uin`：

```javascript
document.cookie.match(/\buin=\D+(\d*)/)
document.cookie.match(/\bilive_uin=\D+(\d+)/)
```

> 表明微云可能在登录后设置 `uin` Cookie（QQ 账号标识）。

### 3.4 关键发现 3 — 无 localStorage-Cookie 双写

微云前端使用传统 jQuery 1.8.3，**没有** SPA 框架（不像腾讯视频的 Vue 3 + `setStorageItem`）。认证体系：

- **TOK** Cookie：会话认证 Cookie（微云专属）
- **fingerprint** Cookie：设备指纹（后端下发）
- **tgw_l7_route**：七层代理路由
- **无 `ams_cookies` 式认证头**：localStorage 不存储认证数据

## 五、设备指纹汇总

| 模块 | 来源 | 隐身模式敏感度 | 说明 |
|:-----|:------|:--------------|:------|
| `fingerprint` Cookie | **服务端**下发 | 🟢 低（持久化 Cookie） | 导入后直接恢复 |
| `AEGIS_ID` (localStorage) | Aegis SDK 前端生成 | 🟡 中（隐身模式下 localStorage 隔离） | 不影响认证 |
| Canvas/WebGL 指纹 | 不存在 | 🟢 不适用 | 微云无前端指纹采集 |

> 微云的设备指纹体系较弱，**隐身/普通模式切换风险低于腾讯视频**。

## 六、跨模式迁移风险评估

| 因素 | 风险评估 | 说明 |
|:-----|:---------|:------|
| Cookie 认证 | ✅ 低风险 | `TOK` 为核心 Cookie，导出后直接可用 |
| localStorage | ✅ 低风险 | `AEGIS_ID` 非认证数据 |
| 设备指纹 | ✅ 低风险 | `fingerprint` Cookie 可导出恢复 |
| 会话 Cookie | ⚠️ 注意 | `TOK` 在未登录态下可能为空 |
| Aegis 监控 ID | 🟢 极低风险 | 仅影响监控上报 |

## 八、结论

腾讯微云使用腾讯 SSO 体系（`ptlogin2.qq.com`），技术栈为 TSW + TAPISIX + stgw + 传统 jQuery（非 SPA）。认证基于 `TOK` Cookie。

关键发现对比：

| 维度 | 腾讯视频 | 腾讯微云 |
|:-----|:---------|:---------|
| 前端框架 | Vue 3 SPA | **jQuery 1.8.3** 传统页面 |
| 双写机制 | ✅ localStorage ↔ Cookie | ❌ 无 |
| 设备指纹 | Canvas/WebGL/Audio 前端采集 | `fingerprint` 服务端下发 |
| Aegis | 未发现 | **AEGIS_ID** localStorage |
| 跨模式迁移 | ⚠️ 高（双写 + 指纹） | ✅ 低风险 |

> 微云的 `fingerprint` Cookie 与腾讯视频的 Canvas 指纹有本质区别——前者是服务端下发的持久化 Cookie（导出即恢复），后者是前端采集的运行时特征（无法通过 Cookie 导出恢复）。因此微云的 Cookie 跨模式迁移风险**远低于**腾讯视频。
