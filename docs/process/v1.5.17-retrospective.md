## v1.5.x 发版问题反思 + v2.0 影响评估

### 一、问题反思

**本轮 v1.5.17 发布暴露了 3 类根因：**

| # | 问题 | 根因 | 已修复 |
|:-:|:----|:----|:-----:|
| 1 | Release 标题反复改（版本号→描述→版本号） | 先发后问，没让用户确认格式 | ✅ `release.sh` 预览功能 |
| 2 | body 重复版本号 / 草稿残留 | 手动拼 curl → 漏参数、忘清理 | ✅ `release.sh` 全自动 |
| 3 | changelog 结构反复改（数组→对象→加title→去掉） | 没有定好数据格式就动手写代码 | ✅ 分类对象 + 双向兼容渲染 |

**已记录到** `docs/process/lessons-learned.md` → **#20**

### 二、对 v2.0 的影响评估

#### ✅ 可直接复用的资产

| 资产 | 说明 |
|:----|:-----|
| `scripts/release.sh` | v2.0 发布时直接跑 `bash scripts/release.sh`，无需手动拼 curl |
| `changelog.json` 分类分组结构 | v2.0 的 changelog 继续沿用，不改格式 |
| `help.js` 双向兼容渲染 | v2.0 如果改 UI 框架，渲染逻辑可直接合并 |

#### ⚠️ 需要提前更新 v2.0 规划文档

当前 `develop:docs/v2.0/development-plan.md` 中 T13 发布流程仍写的是：

```
# 创建 GitHub Release → 上传 zip → 标记 Latest（手动）
```

**建议改为：**
```
bash scripts/release.sh    # 一键发布（自动构建→创建Release→上传附件）
```

另外 T13 的验证步骤（第 1993-1995 行）中的手动检查项，release.sh 已经自动完成了。

#### 📌 架构层面无影响

以上改进集中在**工具链和数据结构**层面，不涉及 v2.0 的三层架构（Layer1 全局 / Layer2 站点 / Layer3 运行时），**不会引入层间耦合**。
