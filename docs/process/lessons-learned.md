# 经验教训与反思

> **更新日期**：持续更新
> **用途**：记录项目开发过程中发现的问题、原因和改正措施，防止同类问题重复发生
> **常态化机制**：每次文档/架构变更后执行「文档健康检查」

## 近期问题复盘（2026-06 项目文档体系建立阶段）

### 问题一：独立文档之间内容不一致

**现象**：
- `development-plan.md` 和 `agent-roles.md` 对 PM 职责描述不一致（一个缺"版本发布"，一个缺"日常决策"）
- 签字链：开发计划仍用旧式 `□` 复选框，而独立文档已更新为 `QA→PD→PM→ME`

**根因**：
1. 文档从开发计划独立出去后，两边各自维护，没有做同步机制
2. 签字链修改时只改了通信拓扑图，忘了更新里程碑签名区

**改正措施**：
- ✅ 建立「文档健康检查清单」（见下文）
- ✅ 每次架构变更后，用自动化脚本扫描所有文档的一致性

---

### 问题二：新文档被孤立（无人引用）

**现象**：
- `agent-roles.md`、`agent-document-matrix.md`、`methodology.md` 三份新文档未被其他 5 份主文档引用
- 从 `PRD.md` 无法跳转到角色定义文档

**根因**：
1. 新增文档后没有检查「谁链接了我」
2. 缺少交叉引用审计机制

**改正措施**：
- ✅ 建立「交叉引用矩阵」自动扫描
- ✅ 新增文档后必须更新所有关联文档的 HEADER 引用行

---

### 问题三：方法论文档残留项目特定内容

**现象**：
- `methodology.md` 的沉淀来源写的是"SessionMaster v2.0 项目开发实践"
- 方法论本应是通用的，却绑定了具体项目名

**根因**：
1. 从项目文档提取方法论时，没有做脱敏检查
2. 没有「通用性检查」环节

**改正措施**：
- ✅ 方法论沉淀完成后必须执行「通用性扫描」——grep 项目特定术语
- ✅ 沉淀来源用"多项目开发实践总结"代替具体项目名

---

### 问题四：标题/数量词与实际情况脱节

**现象**：
- 开发计划标题写"4 代理并行协作"但实际是 5 角色（ME+PM+PD+DE+QA）

**根因**：
1. 架构调整后只改了正文内容，忘了同步 HEADER 元信息
2. HEADER 行容易被忽略——它不在正文滚动视野中

**改正措施**：
- ✅ 架构变更后必须检查所有文档的 HEADER 行
- ✅ 用「版本号+角色数量+层级结构」三元组做一致性检查

## 文档健康检查清单（常态化）

**触发条件**：以下任一情况发生，必须在提交前执行完整检查
- 新增文档
- 修改角色定义或签字权
- 调整组织架构
- 修改通信模式或流程
- 发布新版本前

### 7 项自动检查

| # | 检查项 | 命令/脚本 | 检查什么 |
|:-:|:-------|:----------|:---------|
| 1 | **Agent 输入文档完整性** | 遍历 Agent 定义表，检查每个输入路径存在 | 路径拼写错误、遗漏 |
| 2 | **文档间交叉引用** | grep 每份文档是否引用了其他 7 份 | 孤文档 |
| 3 | **签字链一致性** | 对比 4 个里程碑签名段的角色顺序 | QA→PD→PM→ME 是否统一 |
| 4 | **HEADER 元信息同步** | 版本号、角色数、层级数三值一致 | 标题与正文不脱节 |
| 5 | **术语统一性** | 模式代号（A~G）、角色名（ME/PM/PD/DE/QA）全文档一致 | 术语漂移 |
| 6 | **通用性检查** | grep 项目特定名（SessionMaster、v2.0 等）仅出现在应出现的位置 | 方法论文档脱敏 |
| 7 | **MD 内部链接可达性** | 提取 `[text](path)` 并检查目标文件存在 | 死链接 |

### 3 项手动检查

| # | 检查项 | 怎么做 |
|:-:|:-------|:-------|
| 8 | **内容逻辑一致性** | 随机抽 2 份文档，对比同一概念（如 PM 职责）的描述是否一致 |
| 9 | **数据流图与实际一致** | 查看文档中的数据流图，对照实际文件路径验证 |
| 10 | **旧格式残留** | 全文搜索是否还有旧版术语（如"房间"替代"配对"、旧式复选框 `□`）|

### 执行方式

```bash
# 一键执行 7 项自动检查
cd /opt/projects/session-master && python3 scripts/doc-health-check.py
```

脚本位置：`scripts/doc-health-check.py`（在每次文档变更前运行）

## 反思：为什么会积累这么多问题

| 阶段 | 做了什么 | 没做什么 |
|:-----|:---------|:---------|
| 最初 | 所有内容写在 `development-plan.md` 里 | — |
| 第一次分拆 | 创建了通信目录和卡片模板 | ❌ 没建立独立文档的引用 |
| 第二次分拆 | 创建了 `agent-roles.md` 等 3 份独立文档 | ❌ 没做交叉引用审计 |
| 第三次分拆 | 创建了 `methodology.md` | ❌ 没做通用性脱敏 |
| 架构调整 | 改了通信拓扑图和角色定义 | ❌ 没同步更新所有里程碑签名区 |
| 每次提交 | 只关注新增的内容 | ❌ 没全局扫描已存在的文档 |

**核心教训**：新增或修改文档时，不能只看「我改了啥」，必须看「其他文档是否因此需要同步更新」。

**常态化机制**：把文档健康检查清单做成 `scripts/doc-health-check.py` 脚本，每次文档变更前运行。脚本检查不通过，不提交。

## 版本管理策略

> **背景**：v1.0（master）和 v2.0（develop）两个版本并行维护。master 沿用传统开发方式，develop 采用新设计的多 Agent 规范流程。

### 分支策略

| 分支 | 版本 | 开发方式 | 用途 |
|:-----|:----:|:---------|:-----|
| `master` | v1.x | 传统开发（简单直接） | 稳定发布版，日常维护、bug 修复 |
| `develop` | v2.0 | 多 Agent 规范流程（含 PM/PD/DE/QA 角色） | 新版本架构重构 |

### 核心原则

1. **各自独立**：master 和 develop 分别维护，不互相合入
   - bug 修复如果适用于两个版本 → 各自手动移植（非 merge）
   - 如果修复影响架构兼容性 → 各版本独立决定是否采用
   - 过程文档（`docs/process/`）变更需在两个分支都执行一遍

2. **开发方式各异**：
   - **master（传统）**：直接在 master 上修改、提交、发布，无需角色分工
   - **develop（规范）**：使用 PM 派卡 → PD 设计 → DE 开发 → QA 验证 → PM 终签的完整流程

3. **文档隔离**：
   - `docs/process/` → 版本无关，两个分支都保留
   - `docs/v2.0/` → 仅 develop 分支
   - master 的架构信息通过代码注释和 README 维护

### 版本号规则

| 分支 | 格式 | 示例 |
|:-----|:-----|:-----|
| `master` | `v1.x.y`（语义版本） | `1.5.14` → `1.6.0` → `1.7.0` |
| `develop` | `v2.0.0-dev`（开发中） | `2.0.0-dev` → 发布时改为 `2.0.0` |
| 发布后 | develop 合并替换 master | master 替为 `2.0.0`，旧 master 存档为 `v1.x` 分支 |

### 版本升级检测

> **设计原则**：扩展内置自动检测新版本，用户无需手动检查。

**检测机制**：

1. **首次启动**：onInstalled 后 10 秒执行首次检测
2. **定时检测**：每 6 小时通过 `chrome.alarms` 自动检查
3. **用户手动**：popup 可调用 `checkUpdate` 消息强制检测
4. **缓存策略**：1 小时内不重复请求（节省流量）

**版本比较**：

- 从 GitHub 读取 `VERSION` 文件（`master` 分支 = 稳定版）
- 语义版本比较（`compareVersions` 函数）：`1.5.14` → `1.6.0` ≥ `1.5.15`
- `-dev` / `-beta` 后缀视为开发版，不触发升级通知

**通知方式**：

| 场景 | 行为 |
|:-----|:-----|
| 发现新版本 | 浏览器角标显示 `NEW` + 系统通知 |
| `develop` 测试版 | 可通过设置面板切换更新源到 `BETA_URL` |
| 更新源切换 | 用户设置 `update_url` 指向 develop 分支 VERSION |
| 升级后 | 角标清除，下次检测到更高版本再提示 |

### 本次遗漏复盘

> **经验**：升级检测功能添加到代码后，漏了更新帮助文档和重建打包文件。

| 漏了什么 | 怎么发现的 | 以后怎么做 |
|:---------|:-----------|:----------|
| 帮助页 FAQ 没写「NEW」角标说明 | 用户提醒排查文档 | 功能变更后检查帮助页是否需要同步更新 |
| zip 包没重建（含旧代码） | 用户问「项目有同步吗，包括打包文件」 | 每次功能变更后执行 `bash scripts/build.sh` |
| 帮助页静态版本号没更新 | 排查文档时顺带发现 | ✅ **已修复：改为从 manifest.json 动态读取** |

**核心教训**：凡是有"改了 X 要记得改 Y"的地方，都说明 Y 应该引用 X 而不是硬编码。版本号只是一个例子——端口号、应用名、URL 等静态值都应该遵循单一数据源原则。

**单一数据源原则**：
- `manifest.json` = 版本号的唯一来源
- `config.js` = 运行时动态读取 manifest.json 的 version
- `help.html` = JS 动态填充，不留静态版本号
- `build.sh` = 构建时验证 VERSION 文件 == manifest.json 版本
- 其他静态值（端口 5789、应用名等）也应逐步改为引用方式

**常态化机制**：每次提交前自检清单（追加到原有 10 项检查后）：

| # | 检查项 | 怎么做 |
|:-:|:-------|:-------|
| 11 | **帮助页同步** | 检查帮助页是否需要更新（FAQ/说明/版本号） |
| 12 | **zip 包重建** | `bash scripts/build.sh` 重建并清理旧包 |
| 13 | **经验教训记录** | 本次是否发现了可以沉淀的经验？ |

## 近期问题复盘（2026-06-22 v2.0 审计阶段）

### 问题五：存储键共用导致配置互相覆盖

**现象**：
- `config.js` 中 `STORAGE_KEYS.SYNC_CONFIG` 和 `STORAGE_KEYS.SERVER_SYNC_CONFIG` 都是 `'cloud_sync_config'`
- P2P 配置（signalUrl、p2pDeviceName）和服务器配置（serverUrl、pairKey、deviceName）存储在同一个 key 下
- 如果用户同时使用了两种同步模式，一方配置可能被另一方覆盖

**根因**：
1. copy-paste 定义 key 时没有在最后一步检查值是否唯一
2. 两个 key 在各自模块（P2P 和服务器同步）中独立使用，运行时没有交叉冲突的表征——只有同时使用两种模式时才会暴露

**改正措施**：
- ✅ 立即修复：`SERVER_SYNC_CONFIG` → `'cloud_server_sync_config'`
- ✅ 新增 STORAGE_KEYS 定义检查清单：每次新键必须确保与所有现有键都不重复
- ✅ master 分支同步修复

### 问题六：工时估算脱离实际代码量

**现象**：
- PRD §7：「约 200 行 popup.js 重写」——实际 2,098 行，偏差 10 倍
- T6「三 Tab 内容重构」估算 2.5h——实际至少 7h
- T8「消息路由」估算 1h——实际 2.5h
- T9「渲染函数化」估算 1h——实际 3.5h

**根因**：
1. 写文档时凭印象估算，没有用 `wc -l` 实际测量
2. 低估了 popup.js 中 innerHTML 拼接 + 事件绑定 + 状态管理的复杂度

**改正措施**：
- ✅ 所有工时估算必须基于 `wc -l` 或 `pygount` 实际读数
- ✅ 开发计划中每个任务的工时备注引用实际行数
- ✅ 全局工时从 12-15h 上调至 24-30h

### 问题七：代码已同步但规划文档未更新

**现象**：
- develop 分支已 cherry-pick 了 master 上全部 8 项 UX 改进（更新横幅、设备状态指示器等）
- 但 v2.0 的 PRD、development-plan、test-plan 均未体现这些功能

**根因**：
1. 功能是先有 master 开发再有 cherry-pick 到 develop，文档没跟上代码
2. 对"分支同步"的理解停留在代码层面，忽略了配套文档也要同步

**改正措施**：
- ✅ 更新 PRD.md §2.5 新增 UX 交互描述
- ✅ 在 PRD 中明确标注"以下功能已在 v1.5.x 实现，v2.0 继承"
- ✅ 后续规范：功能 cherry-pick 后必须检查三份规划文档是否需要更新

### 问题八：缺少文档归口管理

**现象**：
- 审计发现 7 份核心文档没有明确的维护责任人
- PRD 更新没有通知 dev-plan 和 test-plan 的流程
- 变更时依赖人工自觉性，没有约束机制

**根因**：
1. 项目初期文档少，靠记忆管理可行
2. 文档体系扩展后（12 份文档），没有建立治理规则

**改正措施**：
- ✅ 在 `agent-document-matrix.md` 新增「文档归口管理规范」章节（7 份文档的维护责任人 / 审核人 / 更新触发条件 / 生命周期）
- ✅ 定义文档更新流程（起草→审核→发布→通知）
- ✅ 定义跨文档一致性原则（PRD 是源头、dev-plan 必须覆盖 PRD、test-plan 必须覆盖 PRD+dev-plan）
- ✅ 定义版本号对齐规则（draft→beta→stable 各阶段要求）
