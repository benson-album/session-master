## 项目管理（PM Agent）

| 类别 | 文档 | 用途 | 操作 |
|:----:|:-----|:-----|:----:|
| 📖 输入 | `development-plan.md` | **调度依据**：从开发计划中取出任务，拆解为子任务分配给 DE/QA | 读 |
| 📖 输入 | `feasibility-report.md` | **技术约束参考**：了解哪些方案已被排除，避免 DE 走弯路 | 读 |
| 📖 输入 | `src/`（项目源代码） | **项目全景**：了解代码结构，判断任务依赖关系和影响范围 | 读 |
| 📝 产出 | `communication/task-cards/` | 任务分配单：写给 DE 的实施任务和写给 QA 的测试任务 | 写 |
| 📝 产出 | `communication/sign-offs/` | 里程碑签字单：批次完成后汇总交付物清单和验证结果 | 写 |
| 📝 产出 | `communication/coordination/` | 协调记录：局部调整（模式 E）的决策和执行记录 | 写 |

## 功能开发（DE Agent）

| 类别 | 文档 | 用途 | 操作 |
|:----:|:-----|:-----|:----:|
| 📖 输入 | `PRD.md` | **需求来源**：知道要做什么、做到什么程度 | 读 |
| 📖 输入 | `development-plan.md` | **任务细节 + 验收标准**：知道当前任务的具体操作步骤、改哪些文件、改完后怎么验证 | 读 |
| 📖 输入 | `feasibility-report.md` | **技术约束**：知道哪些技术方案已被评估为不可行，避免重复踩坑 | 读 |
| 📖 输入 | `test-plan.md` | **测试用例**：了解代码需要通过哪些测试场景，写代码时心中有数 | 读 |
| 📖 输入 | `test-reports/` | **测试结果**：了解 QA 测试中发现的失败和缺陷，用于修复 | 读 |
| 📝 产出 | `communication/impl-reports/` | 实施报告：记录改了哪些文件、构建是否通过、自检清单、发现的问题 | 写 |
| 📝 产出 | `src/`（代码变更） | 实际的代码修改：新建文件、修改文件、删除文件 | 写 |

## 功能测试（QA Agent）

| 类别 | 文档 | 用途 | 操作 |
|:----:|:-----|:-----|:----:|
| 📖 输入 | `test-plan.md` | **测试用例**：逐条执行测试用例，按步骤操作并记录结果 | 读 |
| 📖 输入 | `PRD.md` | **预期行为**：判断实际表现是否符合需求的标准 | 读 |
| 📖 输入 | `development-plan.md` | **T?-V 验证步骤**：每个任务完成后执行对应的验证步骤表格 | 读 |
| 📖 输入 | `src/`（项目源代码） | **实测对象**：将扩展加载到浏览器中实际操作验证 | 读 |

| 📝 产出 | `communication/defects/` | 缺陷记录：发现 bug 时写入 active.md，修复验证后移入 resolved.md | 写 |

## 执行规则

1. **每个 Agent 执行任务前必须读取其「输入」文档**，缺少上下文导致的问题由该 Agent 自行承担
2. **每个 Agent 执行任务后必须产出其「产出」文档**，未产出视为任务未完成
3. **QA 不得读取 DE 的 impl-reports**——质量独立性原则，QA 只以 PRD + test-plan + 代码为准
4. **PD 不直接参与日常调度**——PD 只在收到审查请求或 PRD 需更新时介入
5. **PM 是所有执行层 Agent 的通信枢纽**——DE/PD/QA 之间不可直接通信
6. **我（Hermes 主代理）不介入日常操作**——只审查里程碑、处理 🟠/🔴 级别的升级
