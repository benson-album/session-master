## 执行规则

1. **每个 Agent 执行任务前必须读取其「输入」文档**，缺少上下文导致的问题由该 Agent 自行承担
2. **每个 Agent 执行任务后必须产出其「产出」文档**，未产出视为任务未完成
3. **QA 不得读取 DE 的 impl-reports**——质量独立性原则，QA 只以 PRD + test-plan + 代码为准
4. **PD 不直接参与日常调度**——PD 只在收到审查请求或 PRD 需更新时介入
5. **PM 是所有执行层 Agent 的通信枢纽**——DE/PD/QA 之间不可直接通信
6. **我（Hermes 主代理）不介入日常操作**——只审查里程碑、处理 🟠/🔴 级别的升级

## 文档归口管理规范

### 7 份核心文档的归口责任

| # | 文档 | 维护责任人 | 审核人 | 更新触发条件 | 生命周期 |
|:-:|:-----|:--------:|:-----:|:-----------|:--------|
| 1 | **PRD.md** | PD（产品设计） | Hermes 主代理 | 新功能需求、需求变更、UX 交互细节补充 | 项目全生命周期，版本迭代时更新 |
| 2 | **development-plan.md** | PM（项目管理） | Hermes 主代理 | 🟡 以上方案变更、工时调整、里程碑增删 | 项目执行期，每次批次前更新 |
| 3 | **test-plan.md** | QA（功能测试） | PD（确认覆盖完整） | PRD 变更、新功能加入、缺陷发现需补充用例 | 项目全生命周期，随 PRD 同步更新 |
| 4 | **feasibility-report.md** | PM（项目管理） | Hermes 主代理 | 架构变化、技术方案调整、新增依赖 | 冻结后只读，架构变更时复审 |
| 5 | **maintainability-analysis.md** | DE（功能开发） | PM | 重大模块拆分或重构完成后更新 | 每轮重构后更新 |
| 6 | **agent-document-matrix.md**（本文档） | Hermes 主代理 | — | Agent 角色或文档结构变化 | 自维护，随角色调整更新 |
| 7 | **methodology.md** | Hermes 主代理 | — | 实践中沉淀新的方法论模式 | 持续积累 |

### 文档更新流程

```
触发条件发生
    │
    ▼
维护责任人起草变更 → 提交审核（@审核人） → 审核通过 → 更新文档 → 通知相关角色
    │                      │
    │                      └ 审核不通过 → 退回修改
    │
    └ 🔴 架构级变更 → 需 Hermes 主代理 + 用户共同决策
```

### 跨文档一致性原则

1. **PRD 是源头**——所有功能需求**必须**先在 PRD 中定义，才能出现在 dev-plan 和 test-plan 中
2. **dev-plan 必须覆盖 PRD**——dev-plan 中的任务列表应能追踪回 PRD 的每个章节
3. **test-plan 必须覆盖 PRD + dev-plan**——每条 PRD 需求至少对应 1 个测试用例
4. **变更传播**——修改 PRD 后，维护责任人应同时检查 dev-plan 和 test-plan 是否需要配套更新
