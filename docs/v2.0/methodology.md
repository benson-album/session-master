# 多 Agent 开发方法论

> **版本**：v1.0
> **用途**：沉淀多 Agent 协作开发的可复用方法论，不限项目、不限技术栈
> **关联**：`agent-roles.md` · `agent-document-matrix.md` · `development-plan.md`
> **沉淀来源**：SessionMaster v2.0 项目开发实践

---

## 什么是多 Agent 开发方法论

这是一套**决策框架**，不是任务清单。它定义了：

- 如何组织 5 个角色（我→PM→PD/DE/QA）协同工作
- 每个阶段该做什么、产出什么、谁签字
- 遇到问题时怎么分级、怎么决策
- 如何保证质量独立性和决策可追溯

---

## 核心理念

| # | 原则 | 含义 |
|:-:|:-----|:------|
| 1 | **职责分离** | 每个 Agent 只有一个主要职责。PM 不写代码，QA 不设计，DE 不自测自己的代码 |
| 2 | **决策可追溯** | 每个影响范围/设计/质量的决策都要写下来。没记录就是没发生 |
| 3 | **渐进承诺** | 早定范围、晚定细节。先知道做什么，再决定怎么做 |
| 4 | **质量不可协商** | QA 有独立签字权。QA 不签字=不能发布 |
| 5 | **变化是常态** | 方法论内置了变化处理机制。意外不是失败 |

## 5 阶段生命周期

```
Phase 1       Phase 2       Phase 3       Phase 4       Phase 5
INCEPTION ──→ DEFINITION ──→ PLANNING ──→ EXECUTION ──→ RELEASE
   │             │             │             │             │
   │ 问题定义     │ PRD +       │ 开发计划 +   │ 5 个 Agent   │ 发布
   │ 范围边界     │ 可行性分析   │ 测试计划     │ 按批次工作   │
   │             │             │             │             │
   ▼             ▼             ▼             ▼             ▼
 [Brief]       [Docs]        [Plan]        [Code]        [Release]
```

### Phase 1: 启动（Inception）

**入口**：有一个问题或想法
**出口**：一段问题描述 + 解决方案假设

**谁做什么**：

| Agent | 做什么 |
|:------|:-------|
| 我（ME）| 引导讨论、定义范围边界、识别风险 |
| PM | 协助范围分析 |
| PD | 理解用户需求、调研现有方案 |
| DE | 提供技术可行性输入 |

**产出**：Project Brief（问题/用户/成功标准/范围/风险）

---

### Phase 2: 定义（Definition）

**入口**：已批准的 Project Brief
**出口**：PRD + 可行性报告 + 测试策略 完成并签字

**谁做什么**：

| Agent | 做什么 |
|:------|:-------|
| PD | **主导**——写 PRD，定义功能/UI/数据模型/API |
| DE | 写可行性评估，逐条标记 ✅🟡❌ |
| QA | 写测试策略，定义测试层级和场景 |
| PM | 协调文档编写、做交叉一致性检查 |

**产出**：

```
docs/v1/
├── PRD.md               PD 产出
├── feasibility-report.md DE 产出
└── test-strategy.md     QA 产出
```

---

### Phase 3: 计划（Planning）

**入口**：已批准的 PRD + 可行性报告
**出口**：可执行的开发计划 + 详细测试计划

**谁做什么**：

| Agent | 做什么 |
|:------|:-------|
| PM | **主导**——任务分解、依赖图、里程碑定义、时间估算 |
| DE | 审查任务估算、补充缺失的技术步骤 |
| QA | 编写详细测试用例、定义每个任务的验证步骤 |
| PD | 审查计划的设计覆盖完整性 |

**产出**：

```
docs/v1/
├── development-plan.md   PM 产出（任务分解 + 里程碑）
└── test-plan.md          QA 产出（详细测试用例）
```

---

### Phase 4: 执行（Execution）

**入口**：已提交的开发计划 + 测试计划
**出口**：全部里程碑签字完成、全部缺陷已解决

**谁做什么**：

| Agent | 做什么 |
|:------|:-------|
| PM | **主导**——调度任务、跟踪进度、做日常决策、向我报告 |
| DE | **主导**——写代码、自检、报告问题 |
| QA | **主导**——执行测试、报告缺陷、签质量 |
| PD | 按需审查设计变更 |
| 我（ME）| 审查里程碑、处理决策升级 |

**执行流程**：

```
每个批次：
  │
  ME 委托给 PM（给定方向）
  │
  每个任务：
  ├─ PM 写 TaskCard → 发给 DE
  ├─ DE 实施 + 自检 → 写 ImplReport → 给 PM
  ├─ PM 审查通过 → 写 TaskCard → 发给 QA
  ├─ QA 执行验证 → 写 TestReport → 给 PM
  ├─ PM 审查通过 → 标记任务完成
  └─ PM 更新 todo 看板
  │
  批次完成：
  ├─ PM 写里程碑签字单 → 提交给我
  └─ 我审查 → 批准或升级
```

---

### Phase 5: 发布（Release）

**入口**：全部里程碑已签、全部缺陷已解决
**出口**：Release 已发布 + Git 已打标签

**谁做什么**：

| Agent | 做什么 |
|:------|:-------|
| PM | 版本号升级、Release Notes、部署协调 |
| DE | 构建产物、验证构建 |
| QA | 最终回归测试、签发布质量 |
| PD | 验证文档/帮助页面已同步 |
| 我（ME）| 终签批准发布 |

## 7 种通信模式

### 任务级（单子任务内）

| 模式 | 名称 | 触发 | 流程 |
|:----:|:------|:-----|:-----|
| **A** | 标准串行 | 正常执行 | PM→DE→PM→QA→PM |
| **B** | 设计问题 | DE 发现设计漏洞 | DE→PM→PD→PM→DE |
| **C** | 缺陷修复 | QA 发现 Bug | QA→PM→DE→PM→QA |
| **D** | 阻塞处理 | DE 无法继续 | DE→PM→我→（你）→PM→DE |

### 计划级（跨任务或跨批次）

| 级别 | 代码 | 影响 | 流程 |
|:----:|:----:|:------|:-----|
| 🔵 **微调** | `SCOPE_PATCH` | 单子任务 | DE 自己处理 |
| 🟡 **局部调整** | `SCOPE_ADJUST` | 2+ 相邻任务 | PM 开协调会（模式 E），向我报备 |
| 🟠 **批次调整** | `SCOPE_CHANGE` | 批次范围/交付物 | PM→我→（你）（模式 F）|
| 🔴 **架构变更** | `SCOPE_REDESIGN` | PRD 级设计决策 | PM→我→PD→我→你→全重启（模式 G）|

### 决策树

```
发现问题
  │
  ▼
PM 判断级别
  │
  ├─🔵 仅当前子任务 → DE 自己修，PM 记录
  │
  ├─🟡 影响相邻任务 → PM 开协调会，向我报备
  │
  ├─🟠 影响批次范围 → PM 向我报告 → 我判断：
  │   ├─ 我能决定 → 直接指示 PM
  │   └─ 需要你决定 → 发决策单（≥2方案+推荐）
  │
  └─🔴 PRD 级决策 → PM 向我报告 → 我主持：
      停止全 Agent → PD 审查 → 架构变更单给你 → 你决策
```

### 决策单格式（模式 F 使用）

```markdown
## 🟠 批次调整 — 需要你决策

| 字段 | 值 |
|:-----|:----|
| 触发任务 | 任务编号 |
| 发现问题者 | DE/QA/PD |
| 问题 | 一句话描述 |
| 影响批次 | 批次编号 |
| 影响交付物 | 交付物编号 |

### 影响分析
- 工时变化：+Xh
- 范围变化：说明
- 风险：无/低/中

### 方案 A（推荐）
描述 + 工时 + 优劣

### 方案 B
描述 + 工时 + 优劣

### 我的建议
推荐方案及理由
```

## 质量门禁

每个阶段进入下一阶段前必须满足的条件：

| 阶段过渡 | 门禁条件 | 谁签 |
|:---------|:---------|:----:|
| 启动 → 定义 | Project Brief 已批准 | 我 |
| 定义 → 计划 | PRD + 可行性 + 测试策略 已签字 | 我 |
| 计划 → 执行 | 开发计划 + 测试计划 已提交 | 我 |
| 执行 → 发布 | 全部里程碑签完、缺陷已解决 | QA + 我 |
| 发布完成 | Release 已发布、已打标签 | 我 |

## 决策可追溯

所有决策按以下规则记录：

| 决策类型 | 记录位置 | 格式 |
|:---------|:---------|:-----|
| 🔵 微调 | DE 的 impl-reports 中记录 | discoveredIssues 字段 |
| 🟡 局部调整 | communication/coordination/ | E-编号.md |
| 🟠 批次调整 | communication/coordination/ | F-编号.md（含决策单）|
| 🔴 架构变更 | communication/coordination/ | G-编号.md（含 PD 审查+决策单）|
| 里程碑签字 | communication/sign-offs/ | M-P编号.md |
| 缺陷 | communication/defects/ | active.md / resolved.md |

**原则**：如果没记录，就是没发生。所有影响范围、设计、质量的决策都必须写下来。

## 适配指南

| 项目规模 | 建议 |
|:---------|:-----|
| 小（1-5 任务）| 省略 Phase 1，跳过 PD 角色，PM 兼设计决策 |
| 中（5-20 任务）| 全方法论，PD 仅在里程碑时介入 |
| 大（20+ 任务）| 全方法论，所有角色全时活跃，所有模式可用 |

| 技术栈 | 调整方式 |
|:-------|:---------|
| Web 应用 | 验证方式改为浏览器测试 |
| 库/包 | 更多单元测试、更少人工 QA |
| 移动端 | 不同的构建/发布流程 |
| 数据管道 | 验证策略改为数据验证而非 UI 测试 |
