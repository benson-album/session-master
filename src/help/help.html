一、概述

SessionMaster（会话大师） 是一款专为解决「单设备登录限制」而生的浏览器扩展。 很多 OA 系统、企业内部平台强制用户只能在一处登录，新登录会踢掉旧会话。

经实际测试，大多数 OA 系统的限制是 纯前端实现 的——服务端其实允许多会话共存， 只是浏览器端的 JS 脚本在检测到其他设备登录后主动退出。

核心功能一览

功能说明难度
🍪 Cookie 手动同步手动导出 Cookie 文本，在另一台电脑导入⭐ 简单
☁️ 云端自动同步两台设备通过中继服务器自动双向同步 Cookie⭐⭐ 中等
🛡️ 踢人请求拦截自动阻止浏览器检测多设备登录的请求⭐ 自动
💓 会话保活定时发送心跳请求防止 Cookie 过期⭐ 简单
💡 核心结论：如果贵司的 OA 和本例中的致远 OA V9.0SP1 一样服务端不踢人， 那么只需要 Cookie 同步即可实现多设备同时在线。

二、🍪 Cookie 手动同步

这是最基础、最通用的方式，适用于任何支持 Cookie 的网站。

2.1 导出 Cookie（电脑 A 操作）

1
打开 OA 并登录确保已登录到目标网站
2
点击插件图标在浏览器工具栏点击 SessionMaster 图标
3
切换到「🍪 会话管理 Tab」确认「当前站点」显示的是你的 OA 域名
4
点击「导出 Cookie」所有会话 Cookie 会自动提取出来
5
复制内容点击「复制」按钮，Copy 到剪贴板
6
发送给电脑 B通过微信/飞书/邮件发给另一台电脑
✅ 导出的内容类似：JSESSIONID=ABC123; SESSION=XYZ789

2.2 导入 Cookie（电脑 B 操作）

1
打开目标网站的登录页不需要登录，停留在登录页即可
2
点击插件 → 导入 Cookie在弹出的输入框粘贴电脑 A 发来的内容
3
点击「执行导入」Cookie 会被写入浏览器
4
刷新页面按 F5 刷新，直接进入系统！
⚠️ 注意：Cookie 有过期时间（通常 OA 的会话有效期 30 分钟到 8 小时）。 过期后需要重新导出导入。可以用「会话保活」功能延长有效期。

三、☁️ 云端自动同步

手动复制粘贴太麻烦？云端自动同步让你一次性配好，之后全自动运行。 两台设备通过同一台中继服务器交换加密后的 Cookie。

🔒 安全说明：Cookie 数据在上传前会使用配对码进行 AES-256-GCM 加密， 服务器只能看到密文，无法解密你的 Cookie 内容。加密过程在你的浏览器本地完成。

3.1 部署同步服务器

你需要一台常开设备（如 NAS、云服务器）来运行同步服务器。

方法一：在测试服务器上运行（推荐测试用）

# 上传 server 文件夹到服务器
scp -r session-master/server user@your-server:/tmp/sync-server

# 登录服务器启动（默认端口 5789）
ssh user@your-server
cd /tmp/sync-server
node server.js

# 如需自定义端口（防止冲突）：
# node server.js 5790
# 或 PORT=5790 node server.js

# 验证是否启动成功
curl http://localhost:5789/api/health
# 返回: {"status":"ok","time":"..."}

方法二：NAS / Docker 长期运行

# 在服务器上创建数据目录
mkdir -p /path/to/sessionmaster-sync
# 把 server.js 传过去，后台运行（默认端口 5789）
nodejs /path/to/sessionmaster-sync/server.js &

# 自定义端口：nodejs /path/to/sessionmaster-sync/server.js 5790
# 或用 PM2 管理进程
npm install -g pm2
pm2 start /path/to/sessionmaster-sync/server.js --name session-sync
pm2 save
💡 注意防火墙：确保服务器防火墙放行了相应端口（默认 5789）。 例如 Linux 上可能需要 sudo ufw allow 5789。 如果改了端口，记得同步修改防火墙规则。

3.3 浏览器端配置

1
打开插件 → 「☁️ 同步」Tab
2
填写服务器地址 如果你在服务器上部署了同步服务，就填服务器的地址和端口，例如 http://你的服务器IP:5789
3
生成配对码点击「生成」按钮，会自动生成 8 位配对码（如 X3K8PQ2M）
⚠️ A 电脑和 B 电脑必须使用 同一配对码
4
填写设备名称如「办公室电脑」、「家里笔记本」，方便区分
5
填写同步域名如 你的oa域名.com。留空则自动同步当前站点
6
选择同步间隔建议 5 分钟。太短增加服务器负担，太长可能 Cookie 已过期
7
保存配置 → 打开开关

3.4 P2P 直连配对（推荐）

无需服务器，两台浏览器直接连接。

1
电脑 A：打开插件 → 「🔗 同步」Tab → P2P 直连模式 → 点击「创建配对」
2
电脑 A 会生成一个 6 位配对码（如 X3K8PQ），把此码告知电脑 B
3
电脑 B：打开插件 → 「🔗 同步」Tab → 点击「加入配对」→ 输入配对码
4
连接成功后：开启「P2P 自动同步」开关，填入要同步的域名，Cookie 直接在浏览器之间点对点传输。
💡 需要网络互通：两台电脑必须在互相能通信的网络上（同一局域网、VPN 或内网穿透）。
P2P 直连走 WebRTC，数据全程加密，不经过任何第三方服务器。

3.5 服务器模式（备选方案）

如果 P2P 直连不可用（如网络限制），可使用中央服务器模式：

1
部署同步服务器（见 3.1 节）在一台常开设备上
2
插件中切换到「🖥️ 服务器模式」，填写服务器地址和配对码
3
两台设备使用相同配对码 → 开启开关 → Cookie 通过服务器中转同步
💡 两种模式区别：
• P2P 直连：浏览器直连，数据不经过服务器（推荐）
• 服务器模式：数据经服务器中转，适合跨互联网、NAT 严格的场景
• 两个模式可共存，互不干扰

3.6 关于信令服务器

P2P 直连模式依赖 信令服务器 来完成两台浏览器之间的握手连接。信令服务器仅负责交换连接元数据（SDP、ICE Candidates），时间极短（几十毫秒），不参与实际数据传输，也无法解密或窃听 Cookie 内容。

是否需要自行部署？

  • 如果你部署了同步服务器（server.js），它已内置信令功能（/api/signal/ 端点），无需额外配置
  • 如果你没有部署服务器、仅使用 P2P 直连模式，则需在插件「信令服务器地址」中填写一个可达的信令服务地址
  • 信令服务器所需资源极低，可部署在免费云平台（如 Render、Railway 免费计划），或使用社区提供的公共信令服务

3.7 查看本机网络地址

配置服务器地址时如需获取本机 IP，插件可自动检测：

1
打开插件 → 「🔗 同步」Tab → 拉到最下面
2
展开「🌐 本机网络地址」→ 点击「获取本机网络地址」
3
插件会列出本机的 IPv4 和 IPv6 地址，点击地址即可复制，方便填入「服务器地址」字段
✅ 验证同步状态：打开插件 → 「🔗 同步」Tab，查看同步记录区域。 若有「已导入新 Cookie」或「最后同步时间」则说明同步正常运行。
⚠️ 关于服务器地址填写：
• 局域网内使用，填 内网 IP 即可
• 跨网络访问，需配置 端口转发（5789 端口）或使用 内网穿透（FRP/DDNS）
• 最简单方案：两台电脑在同一局域网下，直接使用内网 IP

四、🛡️ 踢人请求拦截

很多 OA 系统通过浏览器定时发送 AJAX 请求来检测是否有其他设备登录。 插件内置了 9 条通用拦截规则，自动阻止这类检测。

拦截关键词用途
method=checkSession会话检测
method=secondLogin二次登录检测
loginCheck登录核查
sessionTimeout会话超时检测
forcedOffline强制离线
kickOut踢人请求
singleLogin单设备登录检测
duplicateLogin重复登录检测
conflictLogin登录冲突检测

添加自定义规则

如果 OA 使用的检测关键词不在内置列表中，你可以用浏览器的 F12 → Network 抓包， 找到相关请求的 URL 中的关键词，然后在「拦截 Tab」中添加。

1
按 F12 打开开发者工具 → Network 面板
2
过滤关键词如 session、check、online
3
复制请求 URL 中的特征关键词，填入插件的自定义规则中
💡 注意：拦截规则使用 Chrome 的 declarativeNetRequest API， 不需要注入脚本，性能无影响，即使用户不打开插件页面也自动生效。

五、💓 会话保活

Cookie 会过期。OA 系统的会话通常 30 分钟到 8 小时不等。 会话保活功能通过定时向服务器发送请求，保持会话不失效。

配置方法

1
找到保活 URL使用 F12 → Network 过滤请求，找一个定时请求的 URL。常见的有：
/你的oa系统/保持会话的接口
2
打开插件「🍪 会话管理 Tab」在「会话保活」区域输入 URL 和间隔时间
3
点击「💓 添加保活」插件会在后台定时发送请求。添加后可在列表中看到所有保活规则：
  • ✅ 启用 / ⏸️ 暂停 — 开关按钮，随时暂停某条保活
  • ✕ 删除 — 移除不再需要的保活规则
  • 状态栏显示共几条、活跃几条
💡 保活与自动同步的关系：如果开启了云端自动同步或 P2P 直连同步， 同步过程本身就会携带 Cookie 访问 OA，相当于也在做保活，一般不需要单独配置保活规则。

六、常见问题（FAQ）

Q: Edge 浏览器能用吗？

A: 可以。Edge 也是 Chromium 内核，完全兼容 Chrome 扩展。
安装地址：edge://extensions/ → 开发人员模式 → 加载解压缩的扩展

Q: 同步服务器可以部署在公网吗？

A: 可以。服务器只是一个简单的数据中继，不存储解密后的 Cookie。 如果部署在公网，建议用 Nginx 反代加 HTTPS，或者放在内网通过 FRP/DDNS 暴露。

Q: 同步的数据安全吗？

A: 安全。Cookie 数据使用 AES-256-GCM 加密，密钥来源于配对码，服务器无法解密。 即使服务器被攻破，攻击者拿到的也是密文。

Q: 能同步多个网站的 Cookie 吗？

A: 当前版本支持同步一个主域名。同步多个域名需要在「同步域名」字段中填写你要同步的网站域名。 如果你需要同步多个网站，可以稍后在「自定义规则」中扩展。

Q: 开启同步后电脑会变卡吗？

A: 不会。同步使用 Chrome 的 chrome.alarms API 触发，即使浏览器关闭了插件页面， 后台 Service Worker 也会按间隔自动运行。每次同步的数据量很小（几百字节到几 KB）。

Q: 为什么导出了 Cookie，导入后还是没登录？

A: 可能原因：① Cookie 已过期 ② 导入的域名不匹配 ③ 网站绑定了 IP 或 User-Agent（少见） 建议先「清除该站点 Cookie」再重新导入一次。

Q: 手动同步和云端同步有什么区别？

A: 手动同步需要每次导出→发送→导入，适合临时用一次。 云端同步配置好后自动运行，两台浏览器在后台静默交换 Cookie。

Q: 踢人拦截规则不够用怎么办？

A: 内置了 9 条常用规则。如果不够，按 F12 抓包找到特征关键词，在「拦截 Tab」中添加自定义规则。 你也可以联系我们，把新的关键词加到内置规则中。

七、💡 使用技巧

🚀 极简方案（只用 Cookie 导出/导入）：
如果只是临时需要在另一台电脑上访问 OA：
电脑 A 导出 → 拍照/复制 → 电脑 B 导入 → 刷新
⚡ 最省心方案（云端自动同步）：
如果你有一台常开设备（如 NAS、云服务器）：
部署同步服务器 → 两台浏览器都装插件 → 配置同个配对码 → 打开开关 → 搞定
🔒 安全最佳实践：
• 配对码不要用简单数字，用插件生成的随机码
• 同步服务器建议在内网运行（不暴露到公网）
• 如果必须暴露到公网，前面加一层 Nginx HTTPS 反代
• 不再需要同步时，关闭同步开关
📱 手机上能用吗？
Chrome 移动版不支持扩展。但如果你用 Kiwi Browser（安卓）或 Orion（iOS）， 支持安装 Chrome 扩展，SessionMaster 也能用。
