一、概述

SessionMaster（会话大师） 是一款专为解决「单设备登录限制」而生的浏览器扩展。 很多 OA 系统、企业内部平台强制用户只能在一处登录，新登录会踢掉旧会话。

经实际测试，大多数 OA 系统的限制是 纯前端实现 的——服务端其实允许多会话共存， 只是浏览器端的 JS 脚本在检测到其他设备登录后主动退出。

核心功能一览

功能说明难度
🍪 Cookie 手动同步手动导出 Cookie 文本，在另一台电脑导入⭐ 简单
☁️ 云端自动同步两台设备通过中继服务器自动双向同步 Cookie⭐⭐ 中等
🛡️ 踢人请求拦截自动阻止浏览器检测多设备登录的请求⭐ 自动
💓 会话保活定时发送心跳请求防止 Cookie 过期⭐ 简单
💡 核心结论：如果贵司的 OA 服务端不踢人， 那么只需要 Cookie 同步即可实现多设备同时在线。

二、🍪 Cookie 手动同步

这是最基础、最通用的方式，适用于任何支持 Cookie 的网站。

2.1 导出 Cookie（电脑 A 操作）

1
打开 OA 并登录确保已登录到目标网站
2
点击插件图标在浏览器工具栏点击 SessionMaster 图标
3
切换到「🍪 会话管理 Tab」确认「当前站点」显示的是你的 OA 域名
4
点击「导出 Cookie」所有会话 Cookie 会自动提取出来
5
复制内容点击「复制」按钮，Copy 到剪贴板
6
或者下载文件点击「下载文件」按钮，Cookie 将以 JSON 文件保存
⚠️ 重要：登录状态与 Cookie 完整性
  • 登录前导出：只能获取到语言、设备指纹等基础 Cookie，不含认证信息，导入到另一台设备后仍需登录
  • 登录后导出：包含完整的会话/认证 Cookie（如 MUSIC_U、SESSION、token 等），导入后直接生效，无需再次登录
  • 导出内容自动带域名标识：复制后文本首行会标注 # Domain: music.163.com，粘贴到另一台设备导入时自动识别目标域名，无需手动填写
💡 提示：导出的 Cookie 包含 name、value、domain、path、secure、httpOnly 等完整属性， 可以使用「复制」或「下载文件」两种方式传递到另一台电脑。

2.2 导入 Cookie（电脑 B 操作）

1
打开 OA 登录页在另一台电脑打开相同 OA 系统
2
点击插件图标并切换到「🍪 会话管理 Tab」
3
点击「导入 Cookie」在文本框中粘贴从电脑 A 复制的内容
4
点击「执行导入」或者点「从文件导入」选择 JSON 文件
5
刷新页面即可自动登录，无需输入密码
📌 传输方式：可以用微信/飞书/邮件把 Cookie 文本发送到另一台电脑。 建议传输后立即清除剪贴板，防止 Cookie 泄露。

三、☁️ 云端自动同步

在每台电脑上都需要安装插件，配置好同步服务器的地址或 P2P 配对码，Cookie 就会自动在设备间同步。

3.0 快速开始 — 安装本地同步服务器

如果你只需要在两台电脑间同步 Cookie，最快的方式是在其中一台电脑上运行本地同步服务器。插件提供一键安装脚本，无需手动下载 Node.js 或配置服务器。

🐧 Linux / macOS

curl -fsSL https://raw.githubusercontent.com/BenSongLab/session-master/main/src/scripts/sessionmaster-install.sh | bash

脚本自动安装 Node.js、下载 server.js、配置 systemd（Linux）或 launchd（macOS）开机自启。

🪟 Windows（管理员 PowerShell）

iex ((New-Object Net.WebClient).DownloadString('https://raw.githubusercontent.com/BenSongLab/session-master/main/src/scripts/sessionmaster-install.ps1'))

脚本自动安装 Node.js、注册 Windows 服务、开机自启。

✅ 安装后插件自动发现：安装脚本会在本机启动同步服务器（端口 5789）。插件每次打开时自动检测 localhost:5789，检测到后自动填入同步地址，无需手动配置。状态指示器会显示「✅ 本地同步服务器已发现」。

3.1 选择同步方式

插件支持两种同步模式：

  • 🔗 P2P 直连：两台设备通过信令服务器握手后，WebRTC 直连传输 Cookie，数据不经过服务端。适用于两台设备都在内网或都有公网可达的 NAT。
  • ☁️ 云端同步（服务器模式）：所有设备通过指定的同步服务器中转数据。数据经配对码加密后上传，服务器无法解密。

在插件的「🔗 同步 Tab」顶部切换模式：

提示：两种模式互斥，同一时间只能启用一种。切换前需要先关闭另一种。

3.2 P2P 直连模式

1
设备 A：输入设备名称 → 点击「创建配对」生成 6 位配对码
2
设备 B：输入设备名称 → 点击「加入配对」→ 输入配对码
3
配对成功后状态变为「已连接」，显示对端设备名称
4
开启「P2P 自动同步」开关填写要同步的站点域名，开始自动同步

P2P 同步配置

在「P2P 同步配置」可折叠卡片中，可以配置：

  • 服务器地址：信令服务器地址（必须填写）
  • 主从设备模式：可选「主设备」负责上传 Cookie，「从设备」仅接收不下发

3.3 服务器模式

适用于多设备（3 台以上）、跨网络场景。

3.3.2 部署服务端（任选一种）

方式一：Node.js 直接运行（推荐，零依赖）

# 确保安装了 Node.js 18+
node server.js

# 自定义端口
node server.js 5789
# 或: PORT=5789 node server.js

程序监听 0.0.0.0:5789，数据存储在 ./data/ 目录。

方式二：Docker 运行。

docker run -d \
  --name sessionmaster-sync \
  -p 5789:5789 \
  -v ./data:/app/data \
  sessionmaster-sync

3.3.3 浏览器端配置

1
切换到「☁️ 云端同步」模式点击顶部开关切换
2
展开「同步配置」填写服务器地址、配对码、设备名称、同步站点
3
点击「保存配置」确认是否立即开始同步
4
开启「云端双向同步」开关开始定时自动同步

所有设备使用相同的配对码，数据用配对码 AES-256-GCM 加密后传输。

3.4 信令服务器说明

P2P 模式需要一个信令服务器来帮助两台设备建立直连。信令服务器仅用于握手配对，不传输 Cookie 数据。

信令服务器可以与同步服务器是同一台，因为 server.js 同时提供了信令服务和云端同步服务。

配置方式：

  • 在 P2P 同步配置中的「服务器地址」填入信令服务器地址
  • 格式：http://IP:5789

3.5 本机网络地址

在「同步 Tab」底部有一个「🌐 本机网络地址」可折叠卡片。点击「获取本机网络地址」按钮，插件会通过 WebRTC 检测本机局域网 IP 地址。

用途：
  • 如果你用本机作为同步服务器，这里显示的地址就是其他设备要填的「服务器地址」
  • 点击地址可以直接复制到剪贴板
  • 不影响其他功能使用（即使未打开目标网站也可获取）

四、🛡️ 踢人拦截

插件在后台自动拦截含指定关键字的请求，阻止浏览器将「多设备登录」信号发送给服务器。

使用方法

零配置开箱即用：插件安装后，内置的拦截规则库会自动生效。 当你访问匹配的 OA 站点时，插件会自动加载针对该站点的拦截规则。

自定义规则

1
在插件「🛡️ 拦截 Tab」找到「自定义规则」区域
2
输入 URL 关键词如 checkOnline，点击「添加」按钮
3
或者从 Cookie 列表添加先获取当前站点 Cookie，勾选后加入自定义规则

添加的规则会立即生效，拦截所有包含该关键词的请求（请求不会发送到服务器）。

规则库管理

  • 📤 导出：将当前规则库导出为 JSON 文件
  • 📥 导入：从 JSON 文件导入规则库
  • 🔄 从服务器更新：从远程服务器更新规则库（需配置更新地址）

五、💓 会话保活

很多 OA 系统的 Cookie 有效期很短（30 分钟到 8 小时），即使没人踢你，Cookie 也会自动过期。 「会话保活」功能可以定时发送请求到指定 URL，告诉服务器「我还活着」，从而延长会话有效期。

如何使用

1
打开插件 → 「🍪 会话管理 Tab」
2
在「保活 URL」输入框中输入需要定时访问的地址（或点击 📌 填入当前页面地址）
3
选择间隔时间：可选 5、10、30、60 分钟
4
点击「💓 添加保活」即可添加到列表，会自动开始定时保活
5
管理保活规则：在列表中可单独暂停/启用或删除每条规则
💡 提示：
  • 保活在后台 Service Worker 中运行，即使关闭插件弹窗也会持续执行
  • 保活 URL 必须是目标 OA 系统内的有效地址，如 /main.do?method=keepAlive
  • 建议间隔时间设为 OA 会话过期时间的 1/3，例如 OA 会话 30 分钟过期，则设 10 分钟保活一次

六、常见问题（FAQ）

❓ 插件是否兼容所有 OA 系统？
任何基于 Cookie 认证的网站都适用，不限特定 OA。不过拦截模块的内置规则默认覆盖主流 OA 系统（致远、泛微、蓝凌等）。其他系统可能需要手动添加自定义规则。
❓ 如果服务端真的踢人，Cookie 同步还有用吗？
没用。如果是服务端强制踢人，即使你导入了 Cookie 也会被服务端拒绝。好在大多数 OA 只做了前端检测，服务端实际允许多会话共存。
❓ Cookie 会泄露吗？
手动同步：自行选择传输方式（微信/飞书/邮件），注意安全。云端同步：数据在浏览器本地加密后上传，服务器只能看到密文。P2P 模式：数据通过 WebRTC 加密直连传输，不经过任何服务器。
❓ 需要安装服务器吗？
仅用于手动同步或拦截：不需要。仅用于 P2P 同步：需要一个轻量信令服务器（可与他人共用）。使用云端自动同步：需要部署一个同步服务器（server.js，仅 407 行代码，零 npm 依赖）。
❓ 自动同步多久生效？
配置保存后 1 分钟内开始首次同步。默认间隔 5 分钟，可以在设置中调整（1-60 分钟）。
❓ 两台设备同时修改 Cookie 会冲突吗？
服务器模式采用版本控制：每个 Cookie 有独立时间戳，后修改的覆盖先修改的。P2P 模式平等模式下按最后写入覆盖。主从模式下只有主设备上传，从设备仅接收，不会冲突。

七、💡 使用技巧

🔒 安全最佳实践：
• 配对码不要用简单数字，用插件生成的随机码
• 同步服务器建议在内网运行（不暴露到公网）
• 如果必须暴露到公网，前面加一层 Nginx HTTPS 反代
• 不再需要同步时，关闭同步开关
📱 手机上能用吗？
Chrome 移动版不支持扩展。但如果你用 Kiwi Browser（安卓）或 Orion（iOS）， 支持安装 Chrome 扩展，SessionMaster 也能用。
